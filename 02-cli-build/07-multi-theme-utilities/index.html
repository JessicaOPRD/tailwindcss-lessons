How @variant works

Each @variant declaration maps a prefix name to a CSS selector. When Tailwind encounters a prefixed utility such as ocean:bg-blue-50, it generates a rule scoped to that selector. Four declarations cover the four theme/mode combinations.

Light modes

@variant ocean (.theme-ocean &)

@variant forest (.theme-forest &)

ocean:bg-blue-50 generates .theme-ocean .el { background: #eff6ff }. Specificity: 0,2,0 — one theme class plus one element class.

Dark modes

@variant ocean-dark (.theme-ocean.dark &)

@variant forest-dark (.theme-forest.dark &)

ocean-dark:bg-blue-900 generates .theme-ocean.dark .el { ... }. Specificity: 0,3,0 — the compound selector automatically overrides the base variant without relying on source order.

No @variant dark needed: lesson 05 used @variant dark (.dark &) to reconfigure the built-in dark: prefix. This lesson does not declare a standalone dark variant — dark mode is baked into the compound ocean-dark and forest-dark selectors instead.

Verbosity in practice

Every color-bearing property on every element requires all four variant prefixes. The cost is not visible in the HTML (which uses component classes) — it lives in input.css.

Lesson 06 — token approach

The same .utility-card in lesson 06 carries zero prefixed utilities. Both background and border are covered by two var() calls. All four states are resolved by the token cascade in @layer base. 

.utility-card {
  background-color: var(--color-surface-raised);
  border-color:     var(--color-border);
  /* 0 prefixed utilities */
}
The abstraction ceiling: component classes in @layer components hide the verbosity from the HTML — but it still exists in input.css. Inspect input.css and count the @apply lines in any component; compare with lesson 06's equivalent block.

Variant declarations

@variant forest

@variant forest-dark

Declared once in input.css; usable anywhere — in @apply blocks or directly in HTML markup.

Specificity

0,2,0 — base variants

0,3,0 — dark variants

The compound dark selector gains one extra class (.theme-ocean.dark), raising specificity. Dark values win without needing to be declared last in the stylesheet.

Generated CSS shape

Every prefixed utility produces one descendant rule: 

.theme-ocean .el {
  background: #eff6ff
}
.theme-ocean.dark .el {
  background: #1e3a8a
}

Switching mechanism

JS adds one theme class and optionally .dark to <html>. The variant selectors activate because the element now has a matching ancestor. Identical to lesson 06 — the JS toggle logic is the same.

A key limitation: no subtree scoping

Variant selectors match any ancestor with the target class — not the nearest one. This prevents the isolated theme previews that lesson 06 achieved using CSS custom property inheritance.

Why scoping breaks: in lesson 06, wrapping a card in <div class="theme-ocean"> scopes its tokens because var(--color-surface) resolves from the nearest ancestor that defines it.

In this lesson, ocean:bg-blue-50 generates .theme-ocean .el { ... }. If <html> has theme-forest and an inner wrapper has theme-ocean, the element inside matches both .theme-ocean .el and .theme-forest .el — both rules apply, and the last-declared variant wins.

Consequence: the four frozen preview cards from lesson 06 cannot be replicated here without mixing in CSS custom properties. The token cascade approach handles this naturally; selector-based variants do not.
