CLI build · dark mode

Inline dark: utilities

Prefix any Tailwind utility with dark: to apply it when dark mode is active. Because @variant dark (.dark &) is declared in input.css, the .dark class on <html> controls the switch — not the OS setting.

Surface

bg-white

dark:bg-gray-800

The card background itself demonstrates this — white in light mode, gray-800 in dark mode.

Foreground

text-gray-900

dark:text-gray-100

Text color inverts to maintain sufficient contrast against the dark surface.

Border

border-gray-200

dark:border-gray-700

A lighter border value keeps the card edge visible against a dark background without adding visual noise.

Combined

A real component needs all three overrides. Each property that changes requires its own dark: prefix — there is no shorthand.

Inline dark: on every property
When to use: inline dark: utilities work well for one-off elements or when dark mode behavior should be immediately visible in the markup. The tradeoff is verbosity — every changing property on every element must carry its own override.

Semantic color tokens

Declare light-mode values on :root and override them inside .dark in @layer base. Components reference var(--color-*) throughout — when the class toggles, every token updates and no component needs a dark: utility.

Surface

--color-surface-raised

Light: #f9fafb (gray-50)
Dark: #1f2937 (gray-800)

Text primary

--color-text-primary

Light: #111827 (gray-900)
Dark: #f9fafb (gray-50)

Text secondary

--color-text-secondary

Light: #6b7280 (gray-500)
Dark: #d1d5db (gray-300)

Border

--color-border

Light: #e5e7eb (gray-200)
Dark: #374151 (gray-700)

When to use: semantic tokens scale well when many components share the same color decisions. Adding a third mode (e.g. high-contrast) requires only a new .high-contrast { --color-* } override block — every component that uses var(--color-*) inherits it automatically.

Choosing an approach

Inline dark: utilities — best for one-off overrides or when the mode-specific values differ significantly from your token set. Dark mode intent is readable in the markup, but each property needs its own prefix and the repetition grows with component count.

Semantic CSS tokens — best when the same color decisions are shared across many components. Component classes stay clean, theming is centralized in @layer base, and future modes are additive.

Both approaches can coexist — use semantic tokens for structural components and inline dark: utilities for context-specific exceptions.
