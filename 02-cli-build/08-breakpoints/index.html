Mobile-first mental model

Tailwind is mobile-first: unprefixed utilities apply at every width. A breakpoint prefix adds a min-width guard — it activates at that width and remains active at all larger sizes above it. Resize the window to see the active range below update.

base (< 480px) xs · 480px+ sm · 640px+ md · 768px+ lg · 1024px+ xl · 1280px+ 2xl · 1536px+

How the prefix cascade works

class="text-sm md:text-base lg:text-lg" sets font-size three times — the last matching rule wins. At 500px only text-sm applies; at 900px both text-sm and md:text-base match, so md:text-base wins; at 1100px lg:text-lg takes over.

Responsive typography & spacing

Any utility accepts a breakpoint prefix — not just layout classes. The heading and body copy below grow with the viewport, and the container padding widens at the same breakpoints.

Responsive heading

Body copy scales from text-sm to sm:text-base at 640px and lg:text-lg at 1024px. The container padding grows from px-4 to sm:px-8 to lg:px-16 at the same breakpoints. Resize the window to observe both changes at once.

Custom breakpoint

Tailwind v4 generates a responsive prefix from any --breakpoint-* variable declared in @theme. The variable name becomes the prefix name.

How --breakpoint-xs was defined

In input.css: 

@theme {
  --breakpoint-xs: 30rem; /* 480px */
}

This makes xs: a valid responsive prefix everywhere that sm: is valid — including xs:grid-cols-2 in the grid above. The built-in breakpoints (sm, md, lg, xl, 2xl) follow the same convention: each is a --breakpoint-* variable in Tailwind's default theme.

Because @theme variables are emitted as CSS custom properties on :root, JavaScript can read --breakpoint-xs (and any other breakpoint) at runtime — demonstrated in section 6 below.

Breakpoints in JavaScript

Use window.matchMedia to react to breakpoint changes in JavaScript. Both examples below track the same lg threshold (1024px / 64rem); only the source of truth differs.

Example A — Hardcoded value

Pass a string directly to matchMedia. Simple and readable, but the breakpoint value is duplicated between CSS and JS — a future change to the token requires editing both places. 

const mql = window.matchMedia('(min-width: 64rem)')

mql.addEventListener('change', (e) => update(e.matches))
update(mql.matches) // run once on page load

Example B — CSS custom property (v4-specific)

Tailwind v4 emits every @theme variable as a CSS custom property on :root. Reading --breakpoint-lg at runtime means the breakpoint value lives in exactly one place — the @theme block. Update the token and both CSS and JS stay consistent automatically. 

const bp = getComputedStyle(document.documentElement)
             .getPropertyValue('--breakpoint-lg')
             .trim()

const mql = window.matchMedia(`(min-width: ${bp})`)

mql.addEventListener('change', (e) => update(e.matches))
update(mql.matches) // run once on page load
