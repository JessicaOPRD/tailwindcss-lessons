How multi-theme tokens work

Both themes share the same token vocabulary — --color-surface, --color-text-primary, --color-accent, and so on. Each theme/mode combination overrides those names with its own values in @layer base. Components reference var(--color-*) throughout; the cascade does the rest.

theme-forest dark

The two-class system: a theme class (theme-ocean or theme-forest) sets the palette; an optional dark class shifts it to the dark variant. Applying both classes to <html> — or to any ancestor element — selects one of the four states.

Token inheritance in action: each wrapper (div.theme-ocean, div.theme-forest.dark, etc.) defines tokens on itself. CSS custom properties cascade into descendants, so every var() inside the card resolves to that wrapper's values — not to <html>'s. No JavaScript or shadow DOM required.

Live token cards

These cards respond to the header toggles. Every color comes from a var(--color-*) reference — inspect the component CSS in input.css and you will find zero dark: or theme-specific utility prefixes anywhere.

Surface tokens

--color-surface

--color-surface-raised

--color-surface-sunken

Page and card backgrounds. Light modes use near-white tints; dark modes use deeply saturated shades of the active theme.

Text tokens

--color-text-primary

--color-text-secondary

--color-text-muted

Typographic hierarchy. All three values adapt to both the active theme and mode — no dark: prefix on any text element.

Accent tokens

--color-accent

--color-accent-foreground

Primary interactive color and the text placed on top of it. The site header uses --color-accent as its background, making theme changes immediately visible.

Border token

--color-border

Shared border and divider color. A single token covers all card edges and dividers. Inspect the frozen cards above — the .theme-card CSS carries no per-theme overrides.

No theme or dark: utilities needed

Open input.css and inspect any component class — none of them contain a dark: or theme-specific utility prefix. The entire theme system lives in four @layer base blocks.

Scaling to more themes: adding a third theme (e.g. theme-sunset) requires one new CSS block in @layer base defining the same token names with new values, plus a .theme-sunset.dark override — zero component changes. Compare this to the inline-utilities approach in lesson 07, where every color-bearing element would need two new utility prefixes per added theme.
